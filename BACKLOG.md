# BACKLOG.md — memharness improvement backlog (dogfood-derived)

Companion to [`DOGFOOD.md`](./DOGFOOD.md), which holds the dogfood **gate** (use it for real, stopping rules, the 2026-06-23 decision window, how to generate belief data). This file is the **improve** half: it turns the friction that daily use surfaces into a prioritized, evidence-backed backlog. Generated 2026-06-16 from four survey lenses (gaps, measurement, live-store, arch) over the repo + the live store; every row cites real `path:line` or live data.

## 1. The operational loop

Each stage names the actual tool/file that plays the role. (Decision/stopping criteria for *whether to keep going* live in `DOGFOOD.md`; this is the *what to fix* loop.)

| Stage | What happens | Where it lives |
| --- | --- | --- |

| **Capture friction** | (a) **usage.log** — every op + counters, no content, `logUsage` `packages/mcp/src/index.ts:13`; (b) **gardener** — SessionStart hygiene pass (fragmentation + code-map bloat), `packages/core/scripts/memory-gardener.mjs`; (c) **staleness bin** — cold-path git pin-drift check, `packages/mcp/src/staleness.ts`; (d) **write-time nudges** — length nag + code-map smell, `packages/mcp/src/server.ts:199-214`. | usage.log, gardener, staleness bin, server nudges. |
| **Measure** | Eval harness replays a synthetic dataset across retrieval/feature arms; reports recall@k by category. | `packages/eval/src/runner.ts` (`CONFIGS:16`, `runEval:110`); `eval-results.json`. |
| **Triage** | Survey friction vs eval numbers; dedupe; sort by impact-then-effort (this file). Edge-case semantics in `EDGE-CASES.md`; phased plans in `STALENESS-SIGNAL.md`. | this file + `EDGE-CASES.md`. |
| **Fix** | Land in `packages/core` (schema/semantics) or `packages/mcp` (surface/instrumentation), pinning invariants in tests (`test/property/*`, `revise.test.ts`, `recall.test.ts`). | `packages/core/src/{memory,sql}.ts`, `packages/mcp/src/*`. |
| **Re-eval** | Add the new arm/metric first (so the win is visible), re-run eval + gardener + staleness bin, confirm usage.log signals move. | `packages/eval/src/runner.ts`, gardener, staleness bin. |

The loop's weak link today is **Measure → Triage**: usage.log counts hits but not relevance, and `runner.ts` records only `hit: boolean` (line 31) over a synthetic dataset — so fragmentation, code-map rot, semantic dupes, and deployment skew have *no metric that would catch a regression*. Section 4 closes that.

## 3. Top 3 next moves

1. **Startup schema-skew signal (S→M).** The live DB is silently at `user_version=4` while code expects 5 — so source-commit pins, the staleness bin, and the gardener's `hasPin` path all sit dormant and unannounced. Making skew loud (and optionally auto-migrating) turns the whole staleness pipeline from inert to live. Cheapest unblocker of everything else.
2. **Harden the code-map nudge to a hard rejection (>1500ch AND `looksLikeCodeMap`) (S).** Highest impact per line: detector already low-FP, ceiling already exists, live store proves the advisory nudge is ignored. Stops the dominant bloat source at write time instead of a week later in the gardener.
3. **Measure recall relevance, not hit counts (M).** Every ranking lever (staleWeight, hybrid fusion, importance) is invisible while `ProbeOutcome` is `hit: boolean`. precision@k / MRR / rank-of-first-hit in `runner.ts` is what makes the rest of this backlog testable and regression-guarded.

## 4. New evals / metrics

- **Ranking quality** (`runner.ts`): `ProbeOutcome` gains `rankOfFirstGold`, `goldHitsInTopK`; compute MRR/precision@k/recall@k per config. Foundation move.
- **`staleWeight` arm + `staleness-resilience` category** in `CONFIGS:16`, gold mixing fresh+stale facts.
- **Subject-fragmentation metrics**: `subject_fragmentation_ratio`, `alias_precision`; mirror gardener `norm` (`memory-gardener.mjs:50`).
- **Code-map rot metric**: tag facts code-map vs semantic via `looksLikeCodeMap` (`server.ts:48`); for pinned code-maps compute freshness; report `code_map_rot_rate`, `stale_reference_rate`.
- **Semantic-dedup metric**: cluster live facts within subject at cosine ~0.85–0.92; report `dedup_recall`/`dedup_precision`.
- **`logUsage` extensions** (`index.ts:13`): `startup{schema_version,expected_version}`; `rejected{reason,length,subject}`; optional `rated{relevant_count,relevant_positions}` on recall; `session_id` on every op; `fact_length` on remember.
- **Schema-parity assertion** in the harness before `runEval`.
- **Offline labeled live-query set**: 10–20 real queries sampled from usage.log, scored hit@k / precision@k / MRR — the bridge from synthetic eval to real dogfood relevance.
